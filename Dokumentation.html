Session 0 — Dialog-Anzeige

Möglichkeiten zur Benutzerinteraktion aus dem Systemkontext · PowerShell 5.1+ · Stand: 2026-05-08

Überblick

Unter Windows Vista und höheren Versionen existiert Session 0 Isolation — ein Sicherheitsmechanismus, der Services und Systemdienste vom interaktiven Benutzer-Desktop trennt. Dies führt zu einer häufigen Herausforderung: Ein PowerShell-Skript, das im Systemkontext (z.B. während des Bootens, vor Benutzeranmeldung oder per Scheduled Task als SYSTEM) läuft, kann nicht einfach Dialoge oder Benutzerinteraktion auf dem Desktop anzeigen.

Diese Dokumentation stellt die wichtigsten Techniken und Tools zusammen, um diese Isolation zu überbrücken und Dialoge vom Systemkontext aus dem Benutzer anzuzeigen.

  • ServiceUI.exe — der Standard für Deployment-Szenarien (MDT, SCCM, Intune).
  • CreateProcessAsUser — programmgesteuerte Erzeugung eines Prozesses in der Benutzer-Session.
  • WTSSendMessage — einfache Nachrichten-Boxen ohne eigene UI-Ressourcen.
  • Toast Notifications — moderne Benachrichtigungen (mit Einschränkungen).
  • PsExec & WinRM — alternative Approaches über Named Pipes oder Remote Remoting.

Problemstellung: Session 0 Isolation

Seit Windows Vista / Windows Server 2008 gilt: Services laufen in Session 0, Benutzer in Session 1, 2, 3, ... Prozesse in Session 0 haben keinen Zugriff auf:

  • Den aktiven Benutzer-Desktop
  • Die Fenster-Station WinSta0 des angemeldeten Benutzers
  • Eingabegeräte (Tastatur, Maus) des Benutzers
  • Die Display-Instanz des Benutzers

Ein Versuch, UI-Elemente (MessageBox, Fenster, Dialoge) direkt aus Session 0 anzuzeigen, führt dazu, dass diese unsichtbar bleiben — das Skript wartet auf Eingabe, die nie erfolgt, und wirkt „aufgehängt".

Dies betrifft typische Szenarien: Systemstart (vor Benutzeranmeldung), Scheduled Tasks als SYSTEM, Software-Deployment über SCCM/Intune, Boot-Scripts.

Lösung 1: ServiceUI.exe (Microsoft Deployment Toolkit)

Was ist ServiceUI.exe?

ServiceUI.exe ist ein ausführbares Programm aus dem Microsoft Deployment Toolkit (MDT), das als „Session-Bridge" fungiert. Es erkennt automatisch die Session des angemeldeten Benutzers und startet ein Kind-Prozess in dessen Kontext, nicht in Session 0.

    deactivate SUI

Charakteristiken

  • Automatische Session-Detection — erkennt die aktive Benutzersession zur Laufzeit.
  • UI-Anzeige — der Kind-Prozess wird in Winsta0\Default (aktiver Desktop) gestartet.
  • Einfache Nutzung — reicht vor dem Befehl voraus.
  • Weit verbreitet — Standard in Enterprise-Deployment-Szenarien.
  • Wartet auf Beendigung — parent wartet auf Prozess-Exit.

Befehlszeile

# Basis-Syntax
ServiceUI.exe <executable> <arguments>

# Beispiel: PowerShell-Skript mit Parametern
ServiceUI.exe powershell.exe -NoProfile -ExecutionPolicy Bypass -File C:\Scripts\ShowDialog.ps1

# Mit Prozess-Referenz (optional, für Fenster-Station)
ServiceUI.exe -process:explorer.exe powershell.exe -NoProfile -File C:\Scripts\MyScript.ps1

Voraussetzungen & Lizenz

  • MDT oder SCCM Installation erforderlich (kostenlos von Microsoft, open-source-freundlich).
  • ServiceUI.exe muss auf der Maschine erreichbar sein.
  • Läuft nur ab Windows Vista+.
ServiceUI ist die empfohlene Lösung für große Deployments und wenn MDT/SCCM bereits im Einsatz sind.

Lösung 2: CreateProcessAsUser (C# in PowerShell)

Code-Beispiel (gekürzt)

Vollständige PowerShell-Funktion mit CreateProcessAsUser (klicken zum Expand) 
function Invoke-ProcessAsUser {
    [CmdletBinding()]
    param(
        [Parameter(Mandatory=$true)][string]$ProcessPath,
        [Parameter(Mandatory=$false)][string]$Arguments = '',
        [Parameter(Mandatory=$false)][string]$WorkingDirectory = $PWD
    )

    $code = @'
using System;
using System.Runtime.InteropServices;

public class ProcessAsUser {
    [DllImport("kernel32.dll", SetLastError = true)]
    public static extern bool OpenProcessToken(IntPtr ProcessHandle, uint DesiredAccess, out IntPtr TokenHandle);

    [DllImport("advapi32.dll", SetLastError = true)]
    public static extern bool DuplicateTokenEx(IntPtr hExistingToken, uint dwDesiredAccess,
        IntPtr lpThreadAttributes, int ImpersonationLevel, int TokenType, out IntPtr phNewToken);

    [DllImport("kernel32.dll", SetLastError = true)]
    public static extern bool CreateProcessAsUser(IntPtr hToken, string lpApplicationName,
        string lpCommandLine, IntPtr lpProcessAttributes, IntPtr lpThreadAttributes,
        bool bInheritHandles, uint dwCreationFlags, IntPtr lpEnvironment, string lpCurrentDirectory,
        STARTUPINFO lpStartupInfo, out PROCESS_INFORMATION lpProcessInformation);

    [DllImport("kernel32.dll", SetLastError = true)]
    public static extern IntPtr OpenProcess(uint dwDesiredAccess, bool bInheritHandle, uint dwProcessId);

    [DllImport("kernel32.dll", SetLastError = true)]
    public static extern bool CloseHandle(IntPtr hObject);

    [StructLayout(LayoutKind.Sequential)]
    public struct STARTUPINFO {
        public uint cb;
        [MarshalAs(UnmanagedType.LPStr)]
        public string lpReserved;
        [MarshalAs(UnmanagedType.LPStr)]
        public string lpDesktop;
        [MarshalAs(UnmanagedType.LPStr)]
        public string lpTitle;
        public uint dwX;
        public uint dwY;
        public uint dwXSize;
        public uint dwYSize;
        public uint dwXCountChars;
        public uint dwYCountChars;
        public uint dwFillAttribute;
        public uint dwFlags;
        public ushort wShowWindow;
        public ushort cbReserved2;
        public IntPtr lpReserved2;
        public IntPtr hStdInput;
        public IntPtr hStdOutput;
        public IntPtr hStdError;
    }

    [StructLayout(LayoutKind.Sequential)]
    public struct PROCESS_INFORMATION {
        public IntPtr hProcess;
        public IntPtr hThread;
        public uint dwProcessId;
        public uint dwThreadId;
    }
}
'@

    Add-Type -TypeDefinition $code -ErrorAction SilentlyContinue

    $TOKEN_DUPLICATE = 0x0002
    $TOKEN_QUERY = 0x0008
    $TOKEN_ASSIGN_PRIMARY = 0x0001
    $DESIRED_ACCESS = $TOKEN_DUPLICATE -bor $TOKEN_QUERY -bor $TOKEN_ASSIGN_PRIMARY
    $SECURITY_IMPERSONATION = 2
    $TOKEN_TYPE_PRIMARY = 1
    $NORMAL_PRIORITY_CLASS = 0x20
    $CREATE_NEW_CONSOLE = 0x00000010

    $explorerProc = Get-Process -Name explorer -ErrorAction SilentlyContinue | Select-Object -First 1
    if (-not $explorerProc) {
        Write-Error "explorer.exe nicht gefunden"
        return $false
    }

    $hProcess = [ProcessAsUser]::OpenProcess(0x0400, $false, $explorerProc.Id)
    $hToken = 0
    if (-not [ProcessAsUser]::OpenProcessToken($hProcess, $DESIRED_ACCESS, [ref]$hToken)) {
        [ProcessAsUser]::CloseHandle($hProcess)
        return $false
    }

    $hDupToken = 0
    if (-not [ProcessAsUser]::DuplicateTokenEx($hToken, $DESIRED_ACCESS, [IntPtr]::Zero, $SECURITY_IMPERSONATION, $TOKEN_TYPE_PRIMARY, [ref]$hDupToken)) {
        [ProcessAsUser]::CloseHandle($hToken)
        [ProcessAsUser]::CloseHandle($hProcess)
        return $false
    }

    $startupInfo = New-Object ProcessAsUser+STARTUPINFO
    $startupInfo.cb = [System.Runtime.InteropServices.Marshal]::SizeOf($startupInfo)
    $startupInfo.lpDesktop = "WinSta0\Default"
    $startupInfo.dwFlags = 0
    $startupInfo.wShowWindow = 1

    $procInfo = New-Object ProcessAsUser+PROCESS_INFORMATION

    $cmdLine = "`"$ProcessPath`" $Arguments"
    $success = [ProcessAsUser]::CreateProcessAsUser($hDupToken, $ProcessPath, $cmdLine, [IntPtr]::Zero,
        [IntPtr]::Zero, $false, $NORMAL_PRIORITY_CLASS, [IntPtr]::Zero, $WorkingDirectory,
        [ref]$startupInfo, [ref]$procInfo)

    [ProcessAsUser]::CloseHandle($hToken)
    [ProcessAsUser]::CloseHandle($hDupToken)
    [ProcessAsUser]::CloseHandle($hProcess)
    if ($procInfo.hProcess -ne 0) {
        [ProcessAsUser]::CloseHandle($procInfo.hProcess)
        [ProcessAsUser]::CloseHandle($procInfo.hThread)
    }

    return $success
}

# Anwendung
Invoke-ProcessAsUser -ProcessPath "powershell.exe" -Arguments '-NoProfile -Command "Add-Type -AssemblyName System.Windows.Forms; [System.Windows.Forms.MessageBox]::Show(\"Hallo!\")"'

Lösung 3: WTSSendMessage (Einfache Nachrichten)

Was ist WTSSendMessage?

WTSSendMessage ist eine Windows Terminal Services API (in wtsapi32.dll), die es Services erlaubt, einfache Nachrichten-Boxen auf den Desktop eines Benutzers zu senden. Sie ist speziell für Session 0-Prozesse konzipiert.

Lösung 4: Toast Notifications (Windows 10+)

Das Session 0 Problem bei Toast

Toast Notifications können nicht direkt aus dem SYSTEM-Kontext angezeigt werden. Sie benötigen:

  • Benutzer-Session (Session 1+)
  • User Profile Registry
  • WinRT API Zugriff aus User-Context
Lösung: Nutzen Sie CreateProcessAsUser oder ServiceUI als Wrapper, um ein Zwischen-Skript im User-Kontext zu starten — dieses zeigt dann die Toast an.

Praktische Real-World Szenarien

Szenario 1: Sicherheitsupdate mit Bestätigung

Situation: Beim Systemstart (vor Anmeldung) soll der Benutzer gefragt werden, ob ein kritisches Sicherheitsupdate sofort installiert werden soll.

Problem: Task läuft als SYSTEM, bevor Benutzer angemeldet ist → Session 0 wird überbrücken müssen, benutzer aber auch angemeldet sein. Lösung: Task verzögern (z.B. 2 Minuten nach Anmeldung) oder auf "Benutzer-Anmeldungs-Event" reagieren, dann WTSSendMessage oder ServiceUI. 
# Beispiel mit WTSSendMessage
$response = Send-MessageToUser -Title "Sicherheitsupdate" `
    -Message "Sofort installieren und neu starten?" -Style 4  # Yes/No

if ($response -eq 6) {  # IDYES
    Write-Host "Update wird installiert..."
    # Windows Update starten
    Invoke-CimMethod -ClassName Win32_Product -MethodName Install
}

Szenario 2: Software-Deployment mit Fortschrittsanzeige

Situation: SCCM/Intune deployt eine Anwendung, die Benutzer soll Live-Progress sehen (nicht nur ein "lädt..." Fenster).

Lösung: ServiceUI.exe mit PowerShell-Skript, das eine WPF-Form mit Fortschrittsbalken anzeigt. 
# Im Deployment-Script
ServiceUI.exe powershell.exe -NoProfile -ExecutionPolicy Bypass `
    -File "C:\ProgramData\Deploy\Install-WithProgress.ps1" -AppName "Microsoft 365"

Szenario 3: Wartung-Ankündigung vor Neustart

Situation: Ein automatischer Patch-Tag steht an. 1 Stunde vor Neustart soll Benutzer eine Warnung sehen.

Lösung: Task zur geplanten Zeit, WTSSendMessage mit Countdown. 
$remaining = 60  # 60 Minuten
Send-MessageToUser -Title "Wartung anstehend" `
    -Message "System wird in $remaining Minuten neu gestartet. Speichern Sie Ihre Arbeit!" `
    -TimeoutSeconds 30  # Auto-close nach 30 Sekunden

Szenario 4: Multi-User RDP Umgebung

Situation: Terminal Server mit mehreren angemeldeten Benutzern. Nachricht soll an bestimmten Benutzer oder ALLE gehen.

Lösung: WTSEnumerateSessions loopt durch alle Sessions, sendet an alle (oder spezifische). 
# Nachricht an ALLE Benutzer
$sessions = Get-WTSSessions
foreach ($session in $sessions) {
    if ($session.State -eq 'Active') {
        Send-MessageToUser -SessionId $session.SessionId `
            -Title "Admin Warnung" `
            -Message "System wird gleich neu gestartet"
    }
}

Beispiel 1: WTSSendMessage Hilfsfunktion

Klicken zum Expand 
function Send-MessageToUser {
    [CmdletBinding()]
    param(
        [Parameter(Mandatory=$true)][string]$Title,
        [Parameter(Mandatory=$true)][string]$Message,
        [Parameter(Mandatory=$false)][int]$Style = 0,
        [Parameter(Mandatory=$false)][int]$TimeoutSeconds = 0
    )

    Add-Type -TypeDefinition @'
using System;
using System.Runtime.InteropServices;

public class WTSMessage {
    [DllImport("wtsapi32.dll", SetLastError = true)]
    public static extern bool WTSSendMessage(IntPtr hServer, uint SessionId, string pTitle,
        uint TitleLength, string pMessage, uint MessageLength, uint Style, uint TimeOut, out uint pResponse, bool bWait);

    [DllImport("wtsapi32.dll", SetLastError = true)]
    public static extern IntPtr WTSOpenServer(string ServerName);

    [DllImport("wtsapi32.dll", SetLastError = true)]
    public static extern void WTSCloseServer(IntPtr hServer);
}
'@ -ErrorAction SilentlyContinue

    $hServer = [WTSMessage]::WTSOpenServer("localhost")
    $response = 0

    $success = [WTSMessage]::WTSSendMessage($hServer, 1, $Title,
        [uint]$Title.Length, $Message, [uint]$Message.Length, [uint]$Style,
        [uint]$TimeoutSeconds, [ref]$response, $true)

    [WTSMessage]::WTSCloseServer($hServer)

    return $response
}

# Verwendung
$ans = Send-MessageToUser -Title "Frage" -Message "Weitermachen?" -Style 4 -TimeoutSeconds 30
Write-Host "Benutzer antwort: $ans" # 6=Yes, 7=No

$trigger = New-ScheduledTaskTrigger -AtStartup

# Registriere Task
Register-ScheduledTask -TaskName $taskName -TaskPath $taskPath `
    -Action $action -Trigger $trigger -User "SYSTEM" `
    -RunLevel Highest -Force

Write-Host "Task erstellt: $taskPath$taskName"

public class WTSEnum {

    [DllImport("wtsapi32.dll", SetLastError = true)]
    public static extern bool WTSSendMessage(IntPtr hServer, uint SessionId, string pTitle,
        uint TitleLength, string pMessage, uint MessageLength, uint Style, uint TimeOut,
        out uint pResponse, bool bWait);

$hServer = [WTSEnum]::WTSOpenServer("localhost")
$pSessionInfo = 0
$pCount = 0

if ([WTSEnum]::WTSEnumerateSessions($hServer, 0, 1, [ref]$pSessionInfo, [ref]$pCount)) {
    $sessionSize = [System.Runtime.InteropServices.Marshal]::SizeOf([typeof]WTSEnum+WTS_SESSION_INFO)

    for ($i = 0; $i -lt $pCount; $i++) {
        $session = [System.Runtime.InteropServices.Marshal]::PtrToStructure(
            [IntPtr]($pSessionInfo.ToInt64() + $i * $sessionSize),
            [typeof]WTSEnum+WTS_SESSION_INFO)

        if ($session.State -eq 0) {  # Active
            Write-Host "Sende Nachricht an Session $($session.SessionId) ($($session.pWinStationName))"
            $response = 0
            [WTSEnum]::WTSSendMessage($hServer, $session.SessionId, "Admin Nachricht", 14,
                "Server wird in 15 Minuten neu gestartet. Speichern Sie Ihre Arbeit!", 51,
                0, 0, [ref]$response, $false)
        }
    }

    [WTSEnum]::WTSFreeMemory($pSessionInfo)
}

[WTSEnum]::WTSCloseServer($hServer)

Erweitert: RDP & Multi-Session Szenarien

Das RDP-Problem

Remote Desktop Services (RDP/Terminal Server) verkomplizieren das Thema:

  • Mehrere User-Sessions: Nicht nur Session 1, sondern Session 2, 3, 4... (pro RDP-Benutzer eine Session).
  • Welche Session? Wenn der Task auf Session 0 läuft: Welcher RDP-Benutzer bekommt die Nachricht?
  • Console vs. RDP: Session 0 kann Session 1 (Console) sprechen, aber auch RDP-Sessions (2+)?

Lösungsansätze für RDP

AnsatzBeschreibung
Console-spezifischTask spreche nur Session 1 (lokale Console) an. Reicht für Clients mit local Login.
Alle RDP-SessionsWTSEnumerateSessions loopt durch Session 2, 3, 4... und sendet Nachricht an alle aktiven.
Spezifische RDP-UserNutze WTSQuerySessionInformation, um RDP-Benutzer zu identifizieren, spreche nur bestimmte an.
Named Pipes IPCTask schreibt Nachricht in Named Pipe. Jede Session hat einen Reader-Service, der liest und zeigt an.

Robustes CreateProcessAsUser

function Invoke-ProcessAsUserSafe {
    param(
        [Parameter(Mandatory=$true)][string]$ProcessPath,
        [Parameter(Mandatory=$false)][string]$Arguments = '',
        [Parameter(Mandatory=$false)][int]$MaxRetries = 3,
        [Parameter(Mandatory=$false)][int]$RetryDelayMs = 500
    )

    for ($attempt = 1; $attempt -le $MaxRetries; $attempt++) {
        try {
            Write-Verbose "Versuch $attempt/$MaxRetries"

            $explorer = Get-Process -Name explorer -ErrorAction Stop | Select-Object -First 1
            if (-not $explorer) {
                Write-Warning "explorer.exe nicht gefunden (Benutzer nicht angemeldet?)"
                Start-Sleep -Milliseconds $RetryDelayMs
                continue
            }

            $result = Invoke-ProcessAsUser -ProcessPath $ProcessPath -Arguments $Arguments
            if ($result) {
                Write-Host "Erfolgreich"
                return $true
            }
        }
        catch {
            Write-Warning "Fehler: $_"
            if ($attempt -lt $MaxRetries) {
                Start-Sleep -Milliseconds $RetryDelayMs
            }
        }
    }

    Write-Error "Nach $MaxRetries Versuchen fehlgeschlagen"
    return $false
}

# Verwendung
Invoke-ProcessAsUserSafe -ProcessPath notepad.exe -MaxRetries 5

Troubleshooting & FAQ

Dialog erscheint nicht / Fenster ist unsichtbar

  • Ursache: Session 0 Isolation — Prozess läuft in falscher Session/Fenster-Station.
  • Lösung: Nutze ServiceUI, CreateProcessAsUser oder WTSSendMessage.
  • Check: Läuft explorer.exe? (= Benutzer angemeldet?)

ServiceUI.exe nicht gefunden

  • Lösung: Microsoft Deployment Toolkit installieren oder ServiceUI.exe von anderem System kopieren.
  • Pfad: Normalerweise: C:\Program Files\Microsoft Deployment Toolkit\Templates\ServiceUI\ServiceUI.exe

CreateProcessAsUser schlägt mit "Access Denied" fehl

  • Ursache: Token-Handling braucht Admin-Rechte, oder explorer.exe läuft unter anderem Benutzer.
  • Lösung: Task als SYSTEM mit "Run with Highest Privileges".

WTSSendMessage zeigt nichts

  • Ursache: Keine aktive Benutzersession (bevor Anmeldung) oder Session ist disconnected.
  • Lösung: Task-Trigger auf "Bei Benutzer-Anmeldung" oder mit Verzögerung statt "Beim Start".

Task "hängt" oder wartet auf Input

  • Ursache: Skript wartet auf Benutzereingabe, die nie kommt (UI unsichtbar).
  • Lösung: Timeout setzen (-TimeoutSeconds 30) oder Prozess-Output in Datei umleiten.

Toast Notifications funktionieren nicht

  • Ursache: Skript läuft in Session 0 / SYSTEM-Context.
  • Lösung: Toast erst über ServiceUI/CreateProcessAsUser im User-Context aufrufen.

RDP: Welche Session wird angesprochen?

  • Standard: WTSSendMessage mit SessionId=1 spricht Console (lokalen User) an.
  • RDP-User: Enumerate Sessions mit WTSEnumerateSessions, um alle aktiven zu finden.
